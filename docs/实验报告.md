## 1. 项目创意与问题定义

### 1.1 创意目标

Minecraft Pilot Agent 是一个面向中文 Minecraft Java Edition 玩家的单智能体游戏助手。它结合中文 Wiki 知识检索增强（RAG）、官方配方数据查询和游戏日志分析，帮助玩家在游戏过程中获取准确知识、查询合成配方，并在死亡时获得适合的生存建议。

**核心理念**：模型不编造配方、不伪造 Wiki 来源。确定性计算（配方树遍历、哈希校验、死亡解析）全由代码完成，大模型仅负责任务路由、自然语言理解和结果解释。

### 1.2 问题定义

Minecraft 玩家在日常游戏中面临三个高频需求：
1. **知识查询**：中文 Wiki 信息量庞大，玩家需要快速获取与 Java 26.x 版本相关的准确信息
2. **配方规划**：合成复杂物品（如附魔台）需要递归计算材料树，手工推导繁琐且易错
3. **死亡分析**：玩家死亡后希望即时获得简短的生存建议，避免重复犯错

### 1.3 成功标准

| 标准 | 验证方式 |
|---|---|
| Agent 能根据目标选择至少 2 个工具 | Agent 执行日志 |
| 配方树算法确定性、可测试 | pytest 测试覆盖 |
| 中英文死亡事件正确识别 | 双语测试夹具 |
| 知识回答带版本和来源引用 | RAG 检索结果 |
| 安全停止（预算/轮次/未知工具） | Agent 停止状态 |
| 真实模型连通性可验证 | connectivity smoke test |

## 3. 目标用户与核心使用场景

### 3.1 目标用户

- 中文 Minecraft Java Edition 26.2 玩家
- 需要快速查询合成配方和 Wiki 知识的玩家
- 希望获得生存建议的新手玩家

### 3.2 核心使用场景

**场景一：知识问答**

玩家在游戏中打开 Pilot 网页，输入 `/pilot wiki 附魔台怎么用`。Agent 调用 `wiki_search` 工具从 Qdrant 检索中文 Wiki 片段，DeepSeek 整合后返回带来源引用的回答。

**场景二：合成规划**

玩家输入 `/pilot recipe minecraft:enchanting_table`。Agent 调用 `recipe_query` 工具，后端通过确定性算法计算从黑曜石、钻石、书到附魔台的完整 N 层材料树，返回叶子原料汇总。

**场景三：死亡建议**

玩家在游戏中被僵尸击杀，Pilot 检测到 `latest.log` 中的死亡事件，解析死亡原因（中文或英文），调用 DeepSeek 生成 2-5 句简短建议（如"下次可以带盾牌，或在装备附魔保护后再探索"），通过 WebSocket 推送到网页以气泡形式显示。

**场景四：Agent 对话**

玩家输入 `/pilot 我要做一个附魔台`。Agent 状态机先调用 `wiki_search` 确认物品名称和 ID，再调用 `recipe_query` 查询配方树，最后整合结果用自然语言回答。

## 5. 真实大模型接入方式

### 5.1 模型选择

使用 **DeepSeek v4-flash**（通过 OpenAI 兼容 API 访问）。选择理由：
- 原生支持 Function Calling（Tool Calls）
- 中文理解和生成质量优秀
- 成本低廉（适合课程项目多次调用）
- 提供 OpenAI 兼容端点，无需额外 SDK

### 5.2 接入方式

使用 `httpx` 直接调用 REST API（`POST /v1/chat/completions`），不依赖 OpenAI 官方 SDK，实现轻量封装：

```python

    def __init__(self, *, base_url, api_key, model, timeout_seconds, max_tokens):
        ...
    async def chat(self, messages, *, tools=None, tool_choice="auto"):
        url = f"{self._base_url}/v1/chat/completions"
        payload = {"model": self._model, "messages": messages,
                   "max_tokens": self._max_tokens, "temperature": 0.0}
        if tools: payload["tools"] = tools
        ...
    async def connect_test(self):
        """Smoke test"""
        ...
```

### 5.3 连通性测试

提供专用 CLI 脚本验证模型连通性：

```bash
$ python scripts/test_connectivity.py
Connectivity OK: model=deepseek-v4-flash, latency_ms=850, prompt_tokens=12, completion_tokens=8
```

测试输出包含模型名、延迟、token 用量和回答内容，不输出 API 密钥或内部地址。

### 5.4 输出限制与失败处理

| 场景 | 限制策略 |
|---|---|
| Agent 对话 | 最多 4 轮工具调用，输出约 800 tokens |
| 死亡建议 | 单次调用，输出约 200 tokens |
| 日常总量 | 200,000 tokens/天，超限后停止自动死亡分析 |
| DeepSeek 不可用 | 前台静默失败，错误详情进入 `/admin` |
| API 调用失败 | logger 记录脱敏错误，不重试 |

## 6. 智能体架构与工具调用流程

### 6.1 Agent 状态机

```
received → deciding → tool_running → observing → answered/stopped/failed
```

| 状态 | 说明 |
|---|---|
| `received` | 用户消息已接收 |
| `deciding` | 模型判断需要调用哪个工具或直接回答 |
| `tool_running` | 执行工具（wiki_search / recipe_query / recipe_direct） |
| `observing` | 工具结果返回，模型整合信息 |
| `answered` | 生成最终回答 |
| `stopped` | 达到最大轮次或 token 预算 |
| `failed` | 异常或错误 |

### 6.2 MCP 风格工具契约

项目定义了 3 个 MCP 风格工具，采用 JSON Schema 描述输入参数：

**wiki_search**

```json
{
  "name": "wiki_search",
  "description": "在中文 Minecraft Wiki 知识库中搜索与查询相关的内容。",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {"type": "string", "description": "搜索查询"},
      "top_k": {"type": "integer", "description": "返回结果数量，默认 5", "default": 5}
    },
    "required": ["query"]
  }
}
```

**recipe_query**

```json
{
  "name": "recipe_query",
  "description": "查询物品合成配方树。返回直接配方、递归材料树和叶节点原料汇总。",
  "parameters": {
    "type": "object",
    "properties": {
      "item_id": {"type": "string", "description": "目标物品资源 ID"},
      "quantity": {"type": "integer", "description": "目标数量", "default": 1},
      "max_depth": {"type": "integer", "description": "最大递归深度"}
    },
    "required": ["item_id"]
  }
}
```

**recipe_direct**

```json
{
  "name": "recipe_direct",
  "description": "查询物品的直接合成配方列表，不展开材料树。",
  "parameters": {
    "type": "object",
    "properties": {
      "item_id": {"type": "string", "description": "目标物品资源 ID"}
    },
    "required": ["item_id"]
  }
}
```

### 6.3 工具执行与安全边界

- **工具白名单**：仅 wiki_search、recipe_query、recipe_direct 三个工具可用（`TOOL_WHITELIST` frozenset）
- **未知工具拒绝**：模型若请求未注册工具，返回错误且不执行
- **参数二次校验**：tool_executor 内对参数类型和范围做 Pydantic 验证
- **确定性计算**：配方树遍历、SHA-1 校验、死亡解析均由代码完成，不依赖 LLM
- **LLM 角色限制**：仅负责自然语言理解、工具选择和结果解释

### 6.4 执行日志

Agent 每次运行记录结构化 trace，包含：

```json

 {"step": 0, "state": "tool_running", "event": "tool_invoked",
  "tool_name": "recipe_query", "tool_args_summary": "{\"item_id\": \"minecraft:enchanting_table\"}"},

  "token_usage": {"total_tokens": 3120}}]
```

## 8. conda 与 Docker 环境验证情况

### 8.1 conda 环境说明

**本报告如实声明**：项目使用 pip + .venv 虚拟环境，未使用 conda。原因：
- 项目明确使用 Python 3.12 + pip + .venv 方案
- 依赖通过 `pyproject.toml` 声明，`pip install -e '.[dev]'` 一键安装
- Docker 环境提供完全一致的运行容器，隔离 conda 差异

**替代方案**：如需 conda 路径，可基于 `pyproject.toml` 创建 conda 环境文件。推荐使用 Docker Compose 作为统一运行环境。

### 8.2 Docker 环境验证

**Compose 服务**：
```yaml

services:
  qdrant:
    image: qdrant/qdrant:v1.15.3
    ports: ["127.0.0.1:6333:6333"]
    volumes: [qdrant-data:/qdrant/storage]
  app:
    build: .
    ports: ["127.0.0.1:8000:8000"]
    depends_on: [qdrant]
    environment:
      MC_PILOT_QDRANT_URL: http://qdrant:6333
```

## 10. 失败路径、安全拒绝与修正记录

### 10.1 Agent 失败路径

| 失败场景 | 处理方式 | 停止状态 |
|---|---|---|
| 未知工具请求 | 拒绝执行，返回 "未知工具: X" | `stopped` |
| 工具参数无效 | 校验失败，返回错误原因 | `stopped` |
| 达到最大工具轮数 (4) | "已达到最大工具调用次数" | `stopped` |
| 超出每日 token 预算 | "每日 token 配额已用尽" | `stopped` |
| DeepSeek API 调用异常 | 返回 "内部错误"，不重试 | `failed` |

### 10.2 系统失败路径

| 失败场景 | 处理方式 |
|---|---|
| Qdrant 不可用 | 活性正常，就绪显示 degraded，不崩溃 |
| SQLite 不可用 | 就绪显示 degraded |
| Wiki 采集失败 | 保留上一次可用索引，继续运行 |
| 配方 SHA-1 不匹配 | 拒绝导入，清除临时文件 |
| 游戏日志不可用 | 降级模式：RAG 和配方功能仍可用 |
| DeepSeek 密钥未配置 | 死亡建议静默跳过 |

### 10.3 安全拒绝路径

| 场景 | 机制 |
|---|---|
| LLM 请求删除文件 | 工具白名单拒绝（只有 wiki/recipe 三个工具） |
| 非本机访问 /admin | HTTP 403 "仅允许本机访问" |
| 模型输出包含密钥 | `safe_summary()` 永不返回 secret 值 |
| 超大查询 | Agent 不直接操作文件系统 |

### 10.4 开发过程修正记录

| 阶段 | 问题 | 修正 |
|---|---|---|
| M2 配方解析 | shaped 配方中每种材料被重复计算（按槽位去重而非按 key 去重） | 修改 `_parse_shaped_ingredients` 去重逻辑为按 key 字符 |
| M2 ItemOrm | resource_id unique 约束导致多版本插入冲突 | 改为 (resource_id, version_id) 复合 unique |
| M4 tools | `int(object)` 类型不兼容 | 先 `str()` 转换再 `int()` |
| M5 日志尾随 | 截断检测未重置 offset | 文件变小后重置 offset=0 |
| M6 agent-status 路由 | `/api/api/agent-status` 双前缀 | 修正为 `/api/agent-status` |

## 11. 词元成本、安全风险和隐私保护分析

### 11.1 词元预算控制

| 限制项 | 配置值 | 配置位置 |
|---|---|---|
| Agent 最大工具轮数 | 4 | `agent/loop.py:MAX_TOOL_TURNS` |
| Agent 输出上限 | 800 tokens | `agent/client.py:_max_tokens` |
| 死亡建议输出上限 | 200 tokens | 系统提示词长度控制 |
| 每日累计上限 | 200,000 tokens | `agent/memory.py:DAILY_TOKEN_LIMIT` |
| 80% 预警 | 160,000 tokens | 在 /admin 展示预警 |
| 超限后行为 | 确定性工具仍可用，自动死亡分析停止 | `AgentLoop.run()` 预算检查 |

### 11.2 Token 追踪

每次 Agent 调用后记录：`prompt_tokens`、`completion_tokens`、`total_tokens`，并汇总到 `ConversationMemory.daily_tokens`。管理端 `/admin/api/llm` 展示累计用量。

### 11.3 安全风险分析

| 风险 | 缓解措施 |
|---|---|
| API 密钥泄露 | `.env` 文件 gitignored；`.env.example` 不含密钥；`safe_summary()` 不返回密钥；admin API 仅显示 `deepseek_configured: true/false` |
| Prompt 注入 | Agent system prompt 明确角色边界；不执行任意命令；工具白名单限制能力 |
| 数据泄露 | 日志 JSON 不记录完整 prompt、完整 Wiki 文本或玩家聊天；游戏日志绝对路径脱敏 |
| 未授权访问 | 所有服务仅绑定 `127.0.0.1`；admin API 强制 loopback 检查 |
| LLM 幻觉 | 配方计算全由确定性代码完成；RAG 证据不足时明确标注"未经知识库验证" |
| 依赖漏洞 | 依赖版本固定（如 `qdrant-client==1.15.1`），无 `>=` 范围 |

### 11.4 隐私保护

| 保护项 | 措施 |
|---|---|
| 玩家名 | 仅在本地进程内存使用，不持久化、不上传 |
| 聊天内容 | 仅 6 轮会话内存，不持久化；`/pilot clear` 可清除 |
| 死亡消息 | 仅解析后的事件（无原始行），单次生成建议后丢弃 |
| 日志内容 | 游戏日志路径在 admin 中显示为相对路径或脱敏路径 |
| 仓库提交 | 不含 `.env`、Minecraft JAR、模型权重、Qdrant 数据、完整 Wiki 原文 |

## 12. AI 编程智能体使用说明

### 12.1 使用的 AI 工具

本项目的代码 100% 由 **OpenCode AI 编程助手**（底层模型为 DeepSeek-V4-Pro）在 Trellis 工作流框架下生成。开发者通过以下方式指导 AI：

1. **需求引导**：首先与 AI 讨论功能需求和架构设计（Trellis Phase 1 Plan）
2. **分步执行**：按 M1-M8 里程碑分步下达命令（如"开始 M2 任务"）
3. **质量裁决**：AI 自动运行 ruff、mypy、pytest 验证；开发者审阅并确认提交
4. **逐轮修正**：发现问题后反馈给 AI，AI 修改代码并重新验证

### 12.2 AI 工作流 (Trellis)

```
Phase 1: Plan    → 需求讨论、研究、PRD 撰写
Phase 2: Execute → AI 实现代码、运行 lint/typecheck/tests
Phase 3: Finish  → 质量验证、规格更新、git 提交
```

## 13. 项目总结与改进方向

### 13.1 项目成果

本项目成功实现了一个完整的单智能体 Minecraft 游戏助手，包含：

1. **中文 Wiki RAG**：从 9 个分类采集知识，BGE 嵌入 + Qdrant 检索，带来源引用
2. **配方数据管线**：Mojang 官方校验下载，SQLite 存储，确定性 N 层配方树
3. **DeepSeek Agent**：3 个 MCP 风格工具，状态机循环，4 轮上限，token 预算
4. **游戏日志监听**：macOS 日志尾随，中英文死亡解析，低打扰建议推送
5. **网页客户端**：聊天 UI、WebSocket 实时通信、死亡气泡、开发者后台
6. **工程质量**：108 个自动化测试，ruff + mypy 0 errors，Docker Compose 部署

### 13.2 已知局限

- 仅支持 Java Edition 26.2，不支持基岩版或其他版本
- 未实现完整 MCP Server（加分项预留）
- 数据构建需离线手动执行（`build_recipes.py` / `build_wiki.py`）
- Fabric Mod（M9）为阶段二规划，尚未实现
- 死亡建议未集成 Wiki 知识或配方上下文
- 缺少会话持久化（重启后记忆丢失）

### 13.3 改进方向

1. **Fabric 26.2 Mod**：游戏内 `/pilot` 指令、本地 Chat HUD 显示、复用后端 HTTP/WebSocket
2. **完整 MCP Server**：标准化工具协议，支持外部客户端连接
3. **Wiki 增量更新**：基于 MediaWiki revision ID 的增量同步
4. **评估样本集**：50+ 中英文 RAG 问答对，评估命中率、答案质量
5. **库存集成**：读取玩家库存，计算配方缺口
6. **多版本支持**：扩展到 26.1、26.3 等稳定版
