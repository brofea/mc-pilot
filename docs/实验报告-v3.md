# 一、实验内容与步骤

## 1.1 项目立意

Minecraft Pilot Agent 是一个面向 Minecraft Java Edition 26.2 中文玩家的本地游戏助手。玩家打开网页就能问 Wiki 知识、查合成配方、看死亡建议，背后由 DeepSeek 大模型驱动的 Agent 负责理解意图和调度工具。

核心理念：配方计算、数据校验、日志解析这些确定性工作全交给代码，大模型只负责理解用户意图、选择工具和润色回答。模型不编造配方，也不伪造 Wiki 来源。

（此处插入截图：终端运行 `git log --oneline --graph -15`，展示提交历史）

## 1.2 开发历程

项目在 OpenCode AI 编程助手（DeepSeek-V4-Pro）的辅助下开发，按 Trellis 工作流 Plan → Execute → Finish 三阶段、九个里程碑（M1-M9）推进，累计 20 余次提交。

M1 搭建 FastAPI + Qdrant + Docker Compose + SQLite 基础骨架。M2 实现 Mojang CDN 下载 / SHA-1 校验 / 配方提取 / N 层配方树。M3 完成 MediaWiki 采集 / BGE 嵌入 / Qdrant 检索。M4 接入 DeepSeek Agent 与 MCP 风格工具契约。M5 实现游戏日志监听与中英文死亡解析。M6 开发聊天 UI 与诊断后台。M7 补全 121 个测试与安全验证。M8 撰写本报告。M9 完成 Fabric 26.2 Mod 源码与构建（加分项）。

（此处插入截图：`docker compose ps` 展示容器状态，以及 `curl /health/ready` 返回 ready）

## 1.3 系统架构

系统分为八个模块，部署在 app + qdrant 两个 Docker 容器中。

Agent 模块封装 DeepSeek 客户端、三个 MCP 风格工具契约、七状态状态机和会话记忆。Recipes 模块从 Mojang CDN 下载 26.2 JAR，校验 SHA-1 后提取配方存入 SQLite，支持 N 层递归配方树。RAG 模块从 zh.minecraft.wiki 采集九类页面，BGE 嵌入后存入 Qdrant，检索用向量 + 标题混合策略。Game 模块尾随 latest.log，解析中英文死亡消息，去重后调用 DeepSeek 生成建议并通过 WebSocket 推送。API 和 Admin 模块提供 HTTP/WebSocket 路由和仅限本机的诊断后台。前端用 Jinja2 模板加原生 JavaScript，零 Node 构建。

技术选型：Python 3.12 + FastAPI（异步、类型安全），Qdrant（高性能向量库），BAAI/bge-small-zh-v1.5（中文嵌入，本地 CPU），DeepSeek v4-flash（原生 Function Calling、成本低），原生 JS 前端（零构建链）。

（此处插入截图：IDE 中展开 src/mc_pilot/ 的目录结构）

## 1.4 大模型接入与智能体

使用 DeepSeek v4-flash，通过 httpx 直接调 POST /v1/chat/completions，不依赖 OpenAI SDK。Agent 对话限 800 tokens、temperature 0.0；死亡建议限 200 tokens；每日 20 万 token 预算，超限停止 LLM 调用但确定性工具不受影响。API 失败不重试，仅记录脱敏错误。

连通性测试脚本 `scripts/test_connectivity.py` 输出模型名、延迟和 token 用量，不含密钥。

（此处插入截图：`docker compose exec app python scripts/test_connectivity.py` 输出）

Agent 状态机含七个状态：received → deciding → tool_running → observing → answered / stopped / failed，最多四轮工具调用。

定义了三个 MCP 风格工具（JSON Schema 在 agent/tools.py 中）：wiki_search 检索 Wiki 片段并附来源 URL；recipe_query 计算 N 层配方树汇总叶子原料；recipe_direct 只查直接配方。工具白名单 frozenset 拒绝任何未注册工具，参数做 Pydantic 二次校验。

开发中踩过的坑：as_messages 漏序列化 tool_calls 字段导致 400 Bad Request；工具结果被截断到 200 字符导致循环调用。修复后 Agent 完成"理解 → 选工具 → 执行 → 反思 → 回答"的完整链路。

（此处插入截图：容器日志中可见 Agent 轮次——第一次 POST 200 OK（选工具）→ Qdrant 搜索 → 第二次 POST 200 OK（最终答案））

## 1.5 Wiki RAG 与配方管线

RAG：从 zh.minecraft.wiki MediaWiki API 采集方块、物品、生物等九个分类，过滤基岩版和快照页面。BGE 嵌入输出 512 维向量，文本按段落切块（512 字符、64 字符重叠）。Qdrant staging → live 原子切换，点 ID 用确定性 UUID5 映射，scroll 分页正确处理末页边界。

配方：从 Mojang 版本清单获取 26.2 JAR 的 SHA-1 和下载地址，校验后提取 shaped/shapeless/smelting/stonecutting 四类配方 JSON、标签和中英文语言文件。配方树从目标物品递归展开，遇标签材料自动展开，通过祖先集合检测循环，节点上限 5000。26.2 的 ingredient 字段格式有变，加了 isinstance 兼容列表和字典。

（此处插入截图：`docker compose exec app python scripts/build_recipes.py` 完成输出；`docker compose exec app python scripts/build_wiki.py` 显示 "Wiki index built"）

## 1.6 游戏日志与前端

游戏日志监听：启动时若日志不存在，每五秒自动重扫路径，并提供手动重连按钮。基于 inode + offset 增量尾随，处理轮转和截断。死亡解析支持中英文双语（中文 12 个模式、英文 15 个模式），按本地玩家名过滤，未知死亡用 confidence 0.3 兜底。

前端：ChatGPT 风格布局，侧边栏 + 对话区，marked.js + DOMPurify 渲染 Markdown，WebSocket 实时推送死亡建议气泡，手机端自适应。

（此处插入截图：两张——桌面端完整界面，以及 `curl /api/game-state` 返回 connected 状态）

## 1.7 安全与 AI 协作

安全：工具白名单仅三个工具；HTTP 服务只绑 127.0.0.1；Admin 强制 loopback 检查；密钥通过 .gitignore + .env.example + safe_summary 三层保护；日志 JSON 结构化脱敏。成本控制：每日 20 万 token 上限，仅影响 LLM 调用。

AI 协作：代码由 OpenCode 生成，我的角色是审查关键逻辑、确认测试覆盖、Docker 真实验证、做技术取舍（不用 LangChain、不用前端框架、配方完全确定化）。Trellis 三阶段工作流把"想清楚"和"做出来"分开，减少偏差。

# 四、实验心得

Agent 开发里真正磨人的不是大模型调用，而是消息序列化、工具结果传递、向量库边界行为这些中间环节。单靠单元测试不够，必须在真实容器里跑端到端——容器日志不会骗你。

确定性和不确定性的分工很重要。配方树、SHA-1、死亡解析全交代码，大模型只管理解意图和润色回答。边界画清了，安全问题自然消解——LLM 的工具只有三个，每个都有白名单和参数校验。

跟 AI 协作方面，AI 写框架和测试很快，但处理第三方 API 兼容性时常做错误假设。我的角色是做"what if"追问——字段空了怎么办、格式变了怎么办、启动后日志才出现怎么办——然后让 AI 补防御逻辑和测试。Trellis 把需求讨论和实现分开，偏差小了很多。

遗憾：Wiki 构建在 CPU 上跑嵌入耗时较长，可考虑增量更新；Fabric Mod 未与后端联调；尚未建立 RAG 评估样本集。
