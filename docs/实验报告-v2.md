一、实践目的与环境

本项目选题为 Minecraft Pilot Agent，一个面向中文 Minecraft Java Edition 26.2 玩家的本地游戏助手。项目的核心思路是：配方计算、数据校验、日志解析这类确定性工作全部交给代码，大模型只负责理解用户意图、选择工具和润色回答。换句话说，模型不编造配方，也不伪造 Wiki 来源。

选题主要服务于三个真实场景：一是玩家想查某个物品怎么合成，二是玩家想搜 Wiki 上的中文资料，三是玩家在游戏里死亡后想立刻得到几条有用的生存建议。这三个场景刚好能串起一个完整的智能体工程闭环——工具调用、检索增强、实时数据监听。

开发环境方面，后端用 Python 3.12 配合 FastAPI 框架，向量数据库用的 Qdrant（跑在 Docker 里），嵌入模型选了 BAAI 的 bge-small-zh-v1.5（中文效果好、本地能跑），大模型接的是 DeepSeek v4-flash（通过 OpenAI 兼容接口，原生支持 Function Calling）。配方数据存在本地的 SQLite 里。前端是 Jinja2 模板加上原生 JavaScript，用 marked.js 和 DOMPurify 做 Markdown 渲染，没有引入 React 或 Vue。整个项目用 Docker Compose 编排了两个容器（app + qdrant），可以一键启动。

这里需要说明一下 conda 的问题。评分标准里 conda 是必做路径，但本项目从头到尾用的是 pip + .venv 方案。原因是 Docker 已经提供了与宿主机无关的统一运行环境，等价于 conda 的环境可复现目标。依赖全部声明在 pyproject.toml 里，pip install -e '.[dev]' 一行就能装完。如果需要 conda 路径，可以基于 pyproject.toml 生成 environment.yml。

二、实践内容与步骤

2.1 系统架构

整个系统分为八个模块。Agent 模块负责跟 DeepSeek 交互，里面包含了 DeepSeek 客户端封装、三个 MCP 风格的工具契约、Agent 状态机（received → deciding → tool_running → observing → answered / stopped / failed）、六轮会话记忆和每日 token 预算管理。Recipes 模块负责从 Mojang 官方 CDN 下载 26.2 版本的客户端 JAR，校验 SHA-1，提取配方 JSON 和物品标签，存进 SQLite，再通过递归算法计算 N 层配方树。RAG 模块从中文 Minecraft Wiki 的 MediaWiki API 采集九类页面，分块后用 BGE 嵌入，存入 Qdrant，检索时用向量相似度加标题精确匹配的混合策略。Game 模块负责尾随游戏日志文件，处理日志轮转和截断恢复，解析中英文死亡消息，去重后调用 DeepSeek 生成建议，通过 WebSocket 推送给前端。API 模块提供 HTTP 路由（/api/chat、/api/recipes、/api/game-state 等）和 WebSocket 端点。Admin 模块是一个仅限本机访问的诊断后台。Storage 模块封装了 SQLite 引擎和 Qdrant 健康探针。Web 前端采用侧边栏加对话区的布局，类似 ChatGPT 的界面风格。

2.2 大模型接入

大模型选的是 DeepSeek v4-flash。选它的理由有三个：原生支持 Function Calling（这是做 Agent 的刚需），中文理解和生成质量不错，价格也比较低，适合课程项目反复调用。接入方式是直接通过 httpx 库调 REST API（POST /v1/chat/completions），没有依赖 OpenAI 官方 SDK，代码更轻量。

为了控制调用风险，做了一些限制。Agent 对话每次最多输出 800 个 token，temperature 固定为 0.0（保证结果稳定）。死亡建议场景有单独的客户端实例，输出限制在 200 个 token。每天累计不能超过 20 万 token，超了就停止自动建议，但配方和 Wiki 查询这类确定性工具不受影响。API 调用失败不重试，只在日志里记一条脱敏后的错误信息，前端返回"内部错误，请重试"。

另外写了一个独立的连通性测试脚本 test_connectivity.py，发一条固定提示词给 DeepSeek，验证模型可达，输出包含模型名、延迟和 token 用量，不包含密钥。

2.3 智能体状态机与工具调用

Agent 的状态机一共有七个状态。用户消息到达后进入 received，然后进入 deciding，模型判断需要调哪个工具。如果模型返回了 tool_calls 指令，就进入 tool_running 执行工具，执行完进入 observing，把工具结果喂回给模型再进入 deciding。这个过程最多来回四轮，超过四轮就 stopped。如果模型直接返回了文字回答，就进入 answered。过程中如果 API 抛出异常，就进入 failed。

工具方面定义了三个 MCP 风格的契约，每个都用 JSON Schema 描述了参数类型、范围、默认值和必填项。wiki_search 接收 query 和可选的 top_k，从 Qdrant 里检索中文 Wiki 片段并附上来源 URL。recipe_query 接收 item_id、quantity 和可选的 max_depth，计算配方树并汇总叶子原料。recipe_direct 只查直接配方列表，不递归。

安全上做了工具白名单。Agent 里有个 frozenset，只允许这三个工具名。模型要是请求了不在白名单里的工具，直接返回"未知工具"并拒绝执行。工具参数在 executor 里还会做第二次校验，防止 LLM 传了奇怪的类型过来。

会话记忆方面，最多保留六轮对话，超出就踢掉最早的。记忆里的每条记录都保存了角色、内容和 tool_calls 字段，as_messages 方法负责把这些记录拼成 OpenAI 兼容的消息列表发给 API。

这里有一个踩过的坑。第一版里 as_messages 方法没有把 assistant 消息的 tool_calls 字段序列化回去。表现是：LLM 第一次调用选了工具，工具执行完后，第二次调 API 时 DeepSeek 返回 400 Bad Request。原因是 OpenAI 协议要求把 assistant 之前的 tool_calls 原样带回，但代码里漏了。修复后 Agent 就能正常完成"理解问题 → 选工具 → 执行 → 反思 → 回答"的完整流程了。

还有一个问题：工具返回的内容在 add_tool_result 里被截断到了 200 个字符。Agent 拿到不完整的工具结果没办法判断下一步该干嘛，导致反复调用工具直到达到四轮上限。后来把截断逻辑去掉了，完整的工具结果直接传给 LLM，Agent 一次就能给出最终回答。

2.4 Wiki RAG 管线

Wiki 数据从 zh.minecraft.wiki 的 MediaWiki API 拉取，选了九个跟玩家日常关系最大的分类：方块、物品、生物、群系、游戏规则、附魔、状态效果、结构和命令。基岩版、教育版、快照和版本记录页面被过滤掉了。

文本切块用的比较简单，按段落切，每块最长 512 个字符，相邻两块之间有 64 个字符的重叠。嵌入模型用的是 BAAI 的 bge-small-zh-v1.5，本地 CPU 跑，输出 512 维向量。

Qdrant 这边做了 staging 到 live 的原子切换。数据先写进 mc_wiki_staging 集合，全部写完后用 scroll 逐页把点拷到 mc_wiki_live 集合，最后删掉 staging。这个过程中修复了两个 Qdrant 特有的细节问题。一个是点 ID，Qdrant 要求点 ID 必须是无符号整数或 UUID，不能是字符串。所以把原来的 page_id_chunk_index 格式改成了确定性 UUID5 映射（基于固定命名空间 URL 和域 ID），原始 chunk_id 存在 payload 里。另一个是 scroll 分页，Qdrant 在末页可能返回数据但 next_offset 为 None，如果简单判断 None 就认为结束了，会丢失最后一页，而且更糟的是会重新从 offset=None 开始又来一遍。正确的做法是先处理当前页的数据，再检查 next_offset 是否为 None 来决定停不停。

检索的时候用了混合策略。先做一次向量相似度搜索（阈值 0.3），再做一次标题精确匹配搜索，两类结果按 chunk_id 去重后合并，标题匹配的结果排前面。

2.5 配方数据管线

配方数据的信任根是 Mojang 官方的版本清单。流程是先请求版本清单 API 获取 26.2 release 的元数据 URL，从里面拿到客户端 JAR 的下载地址，下载后校验 SHA-1，缓存到本地。然后从 JAR 里提取 data/minecraft/recipe/ 下的所有 JSON（覆盖 shaped、shapeless、smelting、stonecutting 四种类型），提取 data/minecraft/tags/ 下的标签文件，再提取中英文语言文件。

这里踩过一个坑。Minecraft 26.2 版本里，部分配方数据的数据格式变了——ingredient 字段有时候是个列表（多个可选项），有时候是嵌套的字典。原来的代码假设 ingredient 永远是字典，遇到列表就崩溃了。修复方法是加 isinstance 判断，如果是列表就取第一个元素。

配方树的算法是从目标物品出发递归展开的。每展开一层，先查物品有没有直接配方，有的话就把配方里的材料当作新的目标继续递归。遇到标签材料（比如 #minecraft:planks）会自动展开成标签下的具体物品。循环检测是通过维护一个祖先物品集合来实现的，如果当前物品已经出现在祖先路径里了，就停止递归。最后汇总所有叶子节点，去重后输出每个基础原料的需求数量。节点上限设为 5000，深度可以配置。

2.6 游戏日志监听

游戏日志监听这几个版本经历了不少迭代。最初的问题是启动时只扫描一次日志路径，如果那个瞬间日志文件不存在或者游戏还没运行，就静默放弃了，之后再也不尝试。实际场景里，玩家经常先启动了 Pilot 再打开游戏，这个逻辑完全覆盖不到。

所以做了两个改动。第一，给 GameStateService 加了后台自动重试：启动时如果发现日志不存在，就每五秒重新扫一次 macOS 默认路径和 Docker 挂载路径，扫到了就自动开始监听。第二，加了一个 HTTP 接口 POST /api/game-state/reconnect，前端侧边栏里放了一个"重连游戏日志"按钮，玩家任何时候点一下就能手动触发重新扫描。

日志尾随的实现比较朴素，基于 inode 和 offset 做增量读取。轮转的时候 inode 会变，就重置 offset；截断的时候文件变小了，也重置 offset。

死亡解析支持中文和英文两种语言。中文有 12 个正则模式（从高处摔下、落地过猛、在火焰中冉冉升起、淹死了、被射杀了等等），英文有 15 个。每种模式都要求匹配到的玩家名跟本地玩家名一致，忽略其他玩家的死亡事件。如果所有模式都不匹配但日志行里有明显的死亡关键词（died、killed 之类），就用一个置信度 0.3 的 unknown 类别兜底。

玩家名检测也有个坑。日志行 [Server thread/INFO]: brofea[local:E:xxx] logged in with entity id 后面跟了一堆参数，原来的提取逻辑是按空格拆分后找第一个看起来像玩家名的词，结果总是匹配到 [Server 里的 Server。改成了精确搜索 [local:E: 这个模式来定位玩家名。

还有个时序问题。应用启动时会做一次初始扫描，从最近的 500 行日志里提取玩家名和版本信息。如果这 500 行里正好有历史死亡事件，之前是忽略掉的。现在改成扫描到历史死亡后把它注册到去重集合里，避免后续同一个事件被重复处理。但不在启动时生成死亡建议，因为那时候 WebSocket 还没连上，建议发了也没人收。新发生的死亡才会在 poll 循环里检测到并通过 WebSocket 推送到前端。

2.7 前端界面

界面模仿了 ChatGPT 的风格。左边一条 280 像素宽的侧边栏，右上角是应用标题和版本号，下面依次是"新对话"按钮、快捷命令入口、系统状态指示灯和"重连游戏日志"按钮，最底下是开发者后台链接。右边剩余的空间全是对话区域。

对话区用消息气泡区分用户和 Pilot，用户消息右对齐绿底，Pilot 消息左对齐深灰底。聊天的内容支持 Markdown 渲染——Assistant 的回复经过 marked.js 解析成 HTML，再用 DOMPurify 做安全过滤，最后设到 DOM 里。支持的格式包括标题、表格、代码块、引用、列表、链接等等，基本上日常能用的都有。

输入框放在底部，会自动根据内容扩展高度，Enter 发送、Shift+Enter 换行。发送期间输入框禁用，对话区出现一个三点跳动动画表示等待中。手机上侧边栏会收起来变成一个抽屉，左上角有个汉堡菜单按钮可以切换。

2.8 安全与成本控制

安全方面做了几层防护。Agent 的工具白名单只有三个工具，LLM 就算"想"执行别的操作也没有对应的执行器。参数校验在 tool_executor 里还有第二道关卡。所有 HTTP 服务只绑 127.0.0.1，外网访问不到。Admin 后台强制检查请求来源是本机回环地址，Docker 环境下允许私有 IP 网段。

密钥保护方面，.env 文件在 .gitignore 里，仓库里只有 .env.example（不含真实密钥）。应用内部有个 safe_summary 方法，任何需要展示给前端的配置信息都经过它过滤，API 密钥那一栏只显示配置了还是没配置，不显示实际值。日志全部用 JSON 结构化格式，但不记录用户的聊天内容、完整的 Wiki 文章、游戏日志原文。

成本方面只有一个简单的每日 token 计数器。每次 DeepSeek 返回的 usage 里的 total_tokens 会累加到计数器上，超过 20 万就拒绝新的 LLM 请求。但这个限制只影响需要 LLM 的场景——Agent 自然语言对话和死亡建议。配方查询和 Wiki 搜索这两个确定性工具不受影响。

还有一个小细节是 usage 的归一化。DeepSeek 返回的 usage 对象里偶尔会有非整数的字段（比如嵌套的 completion_tokens_details 对象），如果直接往 int 类型的字典里塞会出问题。所以加了一个过滤函数，只保留值是纯整数且不是布尔类型的字段。

2.9 开发过程中的主要修正

整个开发过程中修了不少问题，挑几个典型的说一下。

Agent 对话 400 错误。表现是用 /pilot 命令发自然语言请求后，网页返回"内部错误"。查容器日志发现第一次 DeepSeek 调用成功了（模型选了工具），第二次调用返回 400 Bad Request。原因是 as_messages 构建消息列表时，assistant 消息的 tool_calls 字段没有序列化到 JSON 里，DeepSeek 看到的上下文不完整。改完后就正常了。

工具结果截断导致循环调用。Agent 每轮都调工具但不给最终答案，四轮后达到上限强制停止。原因是 add_tool_result 把工具返回的内容截断到 200 字符，LLM 拿到的信息不够判断。去掉截断后就好了。

配方 26.2 兼容性。Mojang 在 26.2 里改了 ingredient 的数据格式，extractor 里原来假设一定是 dict，遇到 list 就崩了。加了类型判断和列表取首元素逻辑。

Qdrant 点 ID 格式。build_wiki 跑到一半在 upsert 时报错，因为 Qdrant 不接受 page_id_chunk_index 这种字符串 ID。改用 UUID5 确定性映射解决。

logging 保留字段冲突。build_wiki 在 logger.info 的 extra 里用了 name 作为键名，覆盖了 LogRecord.name，Python 直接抛 KeyError。改成 collection_name 就好了。

游戏日志不自动重连。启动时扫一次就放弃，后续游戏打开了也不会重试。加了 5 秒重试循环和手动重连按钮。

玩家名检测误匹配。总是显示 Server 而不显示真实玩家名。原因是 extract_player_from_log 的第三策略在 "logged in with entity id" 行里匹配到了 [Server 里的 Server。改成找 [local:E: 模式。

死亡通知丢失。应用启动时的初始扫描如果检测到历史死亡，会生成建议并尝试推送，但那时候 WebSocket 还没连上，消息就丢了。改为初始扫描只注册去重，不生成建议。

这个项目的代码在 OpenCode 的辅助下生成。工作方式是先跟 AI 讨论清楚需求和设计，让它把 PRD 写出来，然后它按 PRD 一行行写代码，写完自动跑 ruff、mypy 和 pytest 检查，有问题当场改。我主要做的事情是审查关键逻辑（特别是跟外部 API 交互的部分）、确认测试覆盖了边界场景、在 Docker 容器里真实验证每个步骤能不能跑通，以及决定一些技术选型上的取舍——比如不用 LangChain 自己写 Agent 循环、前端不引入框架用原生 JS、配方计算完全确定化不依赖 LLM 等。

3.2 Docker 端到端验收

整个验收流程严格按照 README 里写的步骤执行了一遍。

3.3 Agent 工具调用验证

以"如何合成工作台"为例，Agent 的完整执行路径如下。用户消息到达后，Agent 进入 deciding 状态，DeepSeek 判断需要先查 Wiki 确认工作台的物品 ID 和合成方法，于是返回了 wiki_search 的工具调用指令。工具执行时调用了 BGE 嵌入和 Qdrant 检索，返回了中文 Wiki 中关于工作台的几个段落。Agent 进入 observing 状态，把工具结果加入会话记忆，再次进入 deciding。第二次 DeepSeek 调用整合了 Wiki 信息，直接生成了包含步骤、材料表格和来源链接的自然语言回答。整个过程两次 DeepSeek 调用、一次工具执行、两个主状态转换，没有出现循环或失败。

3.5 安全拒绝路径验证

工具白名单拒绝场景：Agent 收到未注册工具名时，_execute_tool 检查白名单失败，返回包含错误信息的 ToolResult，不会执行任何实际操作。

Admin 访问控制：使用非本机 Host 头请求 /admin/api/status，服务器返回 403，body 内容为"仅允许本机访问"。

四、实践心得

技术层面上最大的体会是，Agent 开发里面真正磨人的不是大模型调用本身，而是那些"看起来不重要"的中间环节。比如消息格式的序列化、工具结果的传递、向量数据库的边界行为、日志文件的轮转处理，每一个忘了检查都可能导致整个链路静默失效。这些坑单靠单元测试不够，必须放到真实容器里跑一遍端到端才能发现。Docker 在这方面的价值远不止"可复现"，它直接改变了验收的质量——你是不是真的跑通了，容器日志不会骗你。

另一个体会是确定性和不确定性的分工很重要。配方树的递归算法、SHA-1 校验、死亡消息的正则解析，这些全部交给代码，没有任何模糊空间。大模型只做它擅长的事情：理解用户想干什么、决定要不要调工具、把结构化数据转成自然的语言。这个边界一旦画清楚了，很多安全问题自然就消解了——LLM 没有文件系统权限、没有执行命令的能力，它的"工具"只有三个，而且每个都有严格的白名单和参数校验。

跟 AI 协作方面，我的感受是 AI 写框架代码和测试确实很快，但在处理第三方 API 兼容性、日志格式差异、外部数据格式变化的场景下，AI 做出的是基于"常见情况"的假设，而这些假设经常被现实打破。所以我的角色更多是在 AI 生成的代码上做"what if"式的追问——如果这个字段空了怎么办、如果 API 返回了意料之外的格式怎么办、如果日志文件在启动后才出现怎么办——然后让 AI 补上对应的防御逻辑和测试。Trellis 工作流把开发拆成 Plan、Execute、Finish 三个阶段对我帮助很大，写 PRD 的过程其实就是把"我想要什么"想清楚，后面 AI 去实现的时候偏差就小了很多。

要说遗憾，一个是 Wiki 索引构建太慢了，在 CPU 上跑几千条向量的嵌入要好一阵，以后可以考虑加增量更新。另一个是 Fabric Mod 虽然源码和构建脚本都有了，但还没跟后端真正联调过。再有就是还没做系统的评估样本集，比如 50 条中英文 RAG 问答对加命中率和质量打分，这个对迭代提示词会很有用。
