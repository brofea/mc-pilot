## 一、实践目的与环境

### 1.1 项目定位

Minecraft Pilot Agent 是一个面向中文 Minecraft Java Edition 26.2 玩家的单智能体游戏助手。核心思路：**确定性计算由代码完成，大模型仅负责自然语言理解和工具选择**。模型不编造配方、不伪造 Wiki 来源。

项目通过三个高频玩家需求验证智能体工程闭环：
1. 中文 Wiki 知识检索增强（RAG）
2. 官方配方数据的确定性 N 层材料树计算
3. 游戏日志实时监听与死亡事件智能建议

### 2.3 真实大模型接入

**模型**：DeepSeek v4-flash，通过 OpenAI 兼容 `POST /v1/chat/completions` 接入，使用 `httpx` 直接调用（不依赖 OpenAI SDK）。

**关键实现** (`src/mc_pilot/agent/client.py`)：

```python
class DeepSeekClient:
    async def chat(self, messages, *, tools=None, tool_choice="auto"):
        """调用 DeepSeek chat completions，支持 Function Calling"""
        payload = {
            "model": self._model,
            "messages": messages,
            "max_tokens": self._max_tokens,
            "temperature": 0.0,
        }
        if tools:
            payload["tools"] = tools
            payload["tool_choice"] = tool_choice
        # ... httpx 异步请求 ...

    async def connect_test(self):
        """Smoke test：发送固定提示词，验证模型可达"""
```

**输出限制与失败处理**：
- Agent 对话：max_tokens=800，temperature=0.0
- 死亡建议：单次调用，max_tokens=200
- 每日总量：200,000 tokens 硬上限
- 超预算：停止自动建议，确定性工具仍可用
- API 错误：logger 记录脱敏信息，不重试，返回"内部错误"

### 2.4 智能体架构与工具调用

#### 2.4.1 Agent 状态机

```
received → deciding → tool_running → observing → answered
                                    ↓                 ↓
                                  deciding ←─────────┘ (有工具结果)
                                                        ↓
                                                   stopped / failed
```

**状态转换规则**：
1. `received`：用户消息到达 → `deciding`
2. `deciding`：模型推理 — 若返回 tool_calls → `tool_running`；若返回文本 → `answered`
3. `tool_running`：执行工具 → `observing`
4. `observing`：工具结果返回 — 若 step < 4 → 重新 `deciding`；若 step ≥ 4 → `stopped`
5. `stopped`：达到 max_tool_turns (4) 或 token 预算超限
6. `failed`：API 异常或未处理错误

**核心代码** (`src/mc_pilot/agent/loop.py`)：

```python
class AgentLoop:
    MAX_TOOL_TURNS = 4

    async def run(self, user_message: str) -> AgentResponse:
        self._memory.add_user(user_message)
        if self._memory.is_over_budget:
            return self._respond("每日 token 配额已用尽", AgentState.stopped)
        return await self._decide()

    async def _decide(self) -> AgentResponse:
        response = await self._client.chat(
            messages=self._memory.as_messages(),
            tools=self._build_tool_schemas(),
        )
        if choice.finish_reason == "tool_calls":
            results = [await self._execute_tool(tc) for tc in tool_calls]
            return await self._handle_observation(results)
        return self._respond(choice.message["content"], AgentState.answered)
```

#### 2.4.3 会话记忆与消息构建

**会话记忆** (`src/mc_pilot/agent/memory.py`)：
- 最多 6 轮对话（user + assistant 各计一轮）
- `AgentTurn` 存储角色、内容、tool_calls 和 tool_call_id
- `as_messages()` 构建 OpenAI 兼容消息列表，正确序列化 assistant 的 `tool_calls` 字段和 tool 消息的 `tool_call_id`
- 每日 token 累计计数，超限拒绝

**修正记录**：第一版中 `as_messages()` 未序列化 assistant 消息的 `tool_calls` 字段，导致 DeepSeek 返回 400 Bad Request。修复后 Agent 可以完成工具调用 → 结果观测 → 反思回答的完整闭环。

### 2.5 中文 Wiki RAG 管线

**数据采集** (`src/mc_pilot/rag/client.py`)：
- 从 `https://zh.minecraft.wiki/api.php` 采集 9 个白名单分类（方块、物品、生物、群系、游戏规则、附魔、状态效果、结构、命令）
- 排除基岩版、教育版、快照和版本记录页面

**嵌入与索引** (`src/mc_pilot/rag/indexer.py`)：
- 文本切块器：按段落切分，最大 512 字符，前后 64 字符重叠
- 嵌入模型：`BAAI/bge-small-zh-v1.5`（512 维），本地 CPU 运行
- Qdrant 存储：staging 集合写入 → 逐页 scroll 迁移 → live 集合原子切换
- 确定性 UUID5 点 ID：`uuid5(NAMESPACE_URL, f"mc-pilot:wiki:chunk:{chunk_id}")`
- 末页 scroll 修复：`next_offset=None` 但有有效记录时仍处理当前页后停止

**检索** (`src/mc_pilot/rag/retriever.py`)：
- Dense 检索（向量相似度）+ Title 精确匹配加权
- 返回结果包含 source URL、revision ID、类别和版本范围
- 分数阈值 0.3（dense），低于阈值的结果丢弃

### 2.6 配方数据管线

**数据来源** (`src/mc_pilot/recipes/downloader.py`)：
- Mojang 官方版本清单 `https://launchermeta.mojang.com/mc/game/version_manifest.json`
- 按版本 ID 查找 26.2 release 的 JAR URL，`piston-meta.mojang.com` 元数据
- 下载后 SHA-1 校验，缓存到 `data/recipes/cache/`

**配方提取** (`src/mc_pilot/recipes/extractor.py`)：
- 从 JAR 中提取 `data/minecraft/recipe/*.json`（shaped/shapeless/smelting/stonecutting）
- 提取 `data/minecraft/tags/*.json`（物品标签，如 `#minecraft:planks`）
- 提取 `assets/minecraft/lang/zh_cn.json` 和 `en_us.json`（本地化）
- 支持 26.2 版本的新数据格式（ingredient 可为列表、嵌套 dict）

**配方树算法** (`src/mc_pilot/recipes/tree.py`)：
- 从目标物品出发递归展开：物品 → 配方 → 材料 → 配方 → …（N 层）
- 标签材料自动展开为具体物品
- 循环检测：同一物品已在祖先路径中出现 → 截断
- 叶子汇总：去重后的基础原材料及各自总需求数量
- 节点上限 5000，深度限制可配置

### 2.7 游戏日志监听与死亡建议

**日志发现** (`src/mc_pilot/game/detector.py`)：
- Docker 模式：读取 `MC_PILOT_GAME_LOG_PATH` 环境变量 → `/minecraft/logs/latest.log`
- 本地模式：默认 `~/Library/Application Support/minecraft/logs/latest.log`
- **自动重试**：首次未找到日志时每 5 秒重试扫描，发现后自动开始监听
- **手动重连**：`POST /api/game-state/reconnect` 端点 + 前端重连按钮

**日志尾随** (`src/mc_pilot/game/tailer.py`)：
- 基于 inode + offset 的增量读取
- 轮转恢复：inode 变化时重置 offset
- 截断恢复：文件变小（新 size < 旧 offset）时重置 offset

**死亡解析** (`src/mc_pilot/game/death_parser.py`)：
- 15 种英文死亡模式（正则匹配 + 玩家名校验）
- 12 种中文死亡模式
- 未知模式 fallback：关键词检测（died/killed/slain/fell/burn/drown 等），confidence=0.3
- 去重：基于 `sha256(player|log_line|timestamp)` 的事件 ID
- 只处理本地玩家死亡，忽略其他玩家

**建议生成**：
- 系统提示词："你是 Minecraft 游戏助手。玩家刚刚死亡，请用 2-5 句简短的话给出建议。不要长篇大论，不要安慰过度，直接给出有用的生存提示。使用中文回答。"
- 用户提示词：`f"玩家 {player} 死亡原因: {raw_message}\n死亡类别: {category}"`
- 单次 API 调用，max_tokens=200
- 通过 WebSocket 推送到网页（右下角红色气泡，12 秒自动淡出）

### 2.9 安全、隐私与成本控制

| 防护层 | 具体措施 |
|---|---|
| **工具白名单** | 仅 wiki_search、recipe_query、recipe_direct 三个工具可用 |
| **参数校验** | tool_executor 对参数类型和范围做二次校验 |
| **输入校验** | chat API 拒绝空消息，JSON body 解析错误返回 422 |
| **路径限制** | Agent 不操作文件系统；配方树目录深度受限 |
| **网络限制** | 所有服务仅 bind `127.0.0.1`，不暴露公网 |
| **Admin 访问控制** | 强制 loopback 检查 + Docker 私有 IP 信任 |
| **密钥保护** | `.env` gitignored；`safe_summary()` 永不返回 secret 值；admin API 仅显示 `deepseek_configured: true/false` |
| **日志脱敏** | JSON 结构化日志不记录完整 prompt、Wiki 文本、玩家聊天、API Key；游戏日志绝对路径脱敏 |
| **Token 预算** | 每日 200,000 tokens 硬上限，超限后确定工具仍可用，LLM 调用停止 |
| **内容净化** | 前端 markdown 渲染通过 DOMPurify 净化，白名单标签和属性 |

### 2.10 开发过程修正记录

| 阶段 | 问题 | 根因 | 修正 | 类型 |
|---|---|---|---|---|
| Agent 对话 | `/pilot <自然语言>` 返回 400 Bad Request | `as_messages()` 未序列化 assistant 的 `tool_calls` 字段 | 添加 tool_calls JSON 序列化逻辑 | 代码缺陷 |
| Agent 对话 | 工具调用循环 4 轮后停止，始终不给出最终回答 | `add_tool_result` 将工具结果截断为 200 字符，LLM 获得的信息不足以判断 | 传递完整工具结果，移除不必要的截断 | 设计缺陷 |
| 配方提取 | 26.2 版新配方格式解析失败 | Mojang 更新了 ingredient 数据结构（列表/嵌套 dict），代码假设总是字典 | 添加 `isinstance(obj, list)` 类型判断和兼容逻辑 | 合同变化 |
| Logging 冲突 | Wiki 构建时 `logger.info(..., extra={"name": ...})` 抛出 KeyError | `extra` 字典中的键 `name` 覆盖了 Python `LogRecord` 保留属性 | 改用 `collection_name` 等域限定的命名 | 接口冲突 |
| Qdrant 点 ID | Wiki 构建在 Qdrant 点 ID 处失败 | Qdrant 不接受 `page_id_chunk_index` 格式的字符串 ID（需 UUID 或整数） | 改为确定性 UUID5 映射，原始 ID 存入 payload | 合同违反 |
| Qdrant 末页 | `swap_to_live` 在 `next_offset=None` 时重新从 offset=None 开始循环 | `scroll` API 在末页返回数据但 `next_offset` 为 None | 处理当前页后检查 None → 跳出循环 | API 误用 |
| 游戏日志 | 玩家名检测提取到 "Server" 而非真实玩家名 | `extract_player_from_log` 第三策略从 "logged in with entity id" 行中误匹配到 `[Server` 线程名 | 精确匹配 `[local:E:` 模式提取玩家名 | 设计缺陷 |
| 游戏日志 | 一次扫描后不再重试 | 初始 `start()` 仅在调用时扫描一次，失败后静默 | 添加 5 秒间隔自动重试 + 手动重连 API + 前端按钮 | 设计缺陷 |
| 死亡通知 | 页面无任何死亡反应 | 初始扫描时 WebSocket 未连接，建议丢失；且历史死亡不被扫描 | 初始扫描注册历史死亡到去重集合，新死亡在 poll 中检测并通过 WebSocket 推送 | 时序问题 |

### 2.11 AI 编程智能体使用说明

本项目代码 100% 由 **OpenCode AI 编程助手**（底层模型 DeepSeek-V4-Pro）在 Trellis 工作流框架下生成。开发流程：

1. **Phase 1 Plan**：与 AI 讨论需求和架构，AI 撰写 PRD 文档
2. **Phase 2 Execute**：AI 按 PRD 实现代码，自动运行 ruff + mypy + pytest 验证
3. **Phase 3 Finish**：AI 更新 spec 文档，开发者审查并确认 git 提交

开发者主动做出的关键判断：
- 不引入 LangChain / LlamaIndex（轻量自建 Agent 循环）
- 配方树全由确定性代码计算，LLM 不参与数量推导
- 使用原生 JavaScript 而非 React/Vue（零构建工具链）
- 不使用 conda（如实声明，Docker 等价可复现）
- Agent 工具白名单 ≤ 3 个（宁可少而安全）

## 四、实践心得

### 4.1 技术收获

1. **Agent 不是简单地"调用 API + 解析结果"**。工具调用的来回通信需要正确处理 OpenAI 的消息格式——assistant 的 `tool_calls` 字段必须序列化回传给模型，否则后端返回 400。这个问题在第一版被遗漏，在 Docker 端到端验证时才暴露。

2. **确定性计算 + LLM 理解的分工是正确策略**。配方树算法全部由代码完成，LLM 只负责路由和解释，避免了模型"算错数量"的风险。这条原则也适用于死亡解析、数据校验等场景。

3. **向量数据库的操作细节决定可靠性**。Qdrant 的 `scroll` API 在末页返回数据但 `next_offset=None`，必须检查记录是否为空而非仅检查 offset。点 ID 必须为 UUID 或整数，不能使用任意字符串。这些细节在单元测试中不易暴露，需要结合真实容器运行验证。

4. **Docker 端到端验收不可替代**。健康检查、静态代码分析、单元测试都不能替代在真实容器中执行 `build_recipes.py`、`build_wiki.py` 并通过 HTTP 请求验证对话链路。READEME 中的命令必须在交付镜像中至少完整执行一次。

### 4.2 AI 协作反思

- AI 编程助手（OpenCode/DeepSeek-V4-Pro）在生成框架代码、编写测试和格式化文档方面效率极高，但**边界条件和第三方 API 兼容性**需要开发者主动思考和验证。
- Trellis 工作流（Plan → Execute → Finish）提供了清晰的分步框架，PRD 驱动的开发方式避免了"想到哪写到哪"的混乱。
- 多轮 bug 修复中存在"修 A 坏 B"的倾向（logging 冲突修复 → scroll 问题暴露 → UUID 问题暴露），说明修改后回归验证不应局限于修改点。

### 4.3 改进方向

1. 完善 MCP Server 实现（当前为 MCP 风格契约，非完整 MCP 协议）
2. Wiki 增量更新（基于 revision ID），避免全量重建
3. 会话持久化（数据库存储），重启后保留记忆
4. 评估样本集（50+ RAG 问答对），用于提示词优化
5. Fabric Mod 集成（M9 源码已预留，需端到端联调）
