# Minecraft Pilot Agent 产品与技术方案

## Goal

设计并最终实现一个面向中文 Minecraft 玩家的单智能体游戏助手。它以中文 Wiki RAG 回答知识问题，以 MCP 风格 Function Calling 查询结构化配方，并能围绕玩家目标形成计划、选择工具、记录过程和安全停止，满足课程综合实验的可运行、可测试、可复现与可解释要求。

## What I already know

- 项目是个人课程综合实验，当前仓库只有两份课程文档，尚无代码或既有技术框架。
- 用户明确只做一个 Minecraft Pilot Agent，不做多智能体系统。
- 功能 1：爬取中文 Wiki 并建立 RAG，回答 Minecraft 知识问题。
- 功能 2：获取配方全集，以 MCP 风格的 Function Calling 供 Agent 查询；不实现真正的 MCP Server。
- 功能 3：实时监听 Minecraft 客户端 `latest.log`；检测玩家死亡事件，依据死亡原因主动给出合理建议。
- 当前交付形态为独立网页；长期目标是演进为游戏内 Mod。
- 本次验收目标暂定固定为 Minecraft Java Edition 26.2 正式版；排除快照、预发布版和候选版。数据结构保留版本字段，但不承诺本次覆盖全部 26.x。
- Wiki 数据源确定为中文 Minecraft Wiki MediaWiki API：`https://zh.minecraft.wiki/api.php`。
- Wiki 收录范围采用 Java 26.x 核心知识分类白名单，不抓取整个主命名空间，也不在问答时临时在线补抓。
- 配方事实源确定为各个 Java 26.x 稳定版的官方原版游戏数据包；中文 Wiki 不作为配方真值源。
- 游戏包获取策略：依据 Mojang 官方版本清单自动下载；官方 CDN 失败时允许使用国内镜像作为传输回退，但仍以官方元数据和 SHA-1 为唯一信任依据。
- 当前游戏版本采用“日志自动识别 + 网页手动覆盖”；未检测到 Minecraft 运行时关闭日志监听，但 Wiki RAG 与配方树仍可使用手动选择的受支持版本。
- 交付顺序确定为两阶段：先完成 Docker 化的可用网页，再实现基于 Fabric API 的 26.2 游戏内 Mod。
- Docker 用于统一后端、Web 与 VectorDB 运行环境；首要保证 macOS，其他系统提供兼容设计与说明。
- Fabric Mod 采用轻量客户端架构，通过 localhost HTTP/WebSocket 调用与网页共用的 Agent 后端。
- 后端技术栈确定为 Python 3.12 + FastAPI + Pydantic；依赖通过 pip 安装到项目隔离 `.venv`，并提供 Docker 镜像。
- 明确禁止安装或使用 conda（主机与项目均不执行 conda 命令）。这与课程文档的 conda 必做要求冲突，作为已知验收偏差记录。
- 生成模型确定为 DeepSeek 在线 OpenAI 兼容 API；API Key 由用户在本地 `.env` 提供，仓库只提交 `.env.example`。
- Wiki embedding 确定使用本地 `BAAI/bge-small-zh-v1.5`，在后端容器内运行，不调用额外在线 embedding API。
- VectorDB 确定使用 Qdrant Docker 服务；主机不通过 Homebrew 或其他方式单独安装 Qdrant。
- Agent 编排与 RAG 管线确定轻量自建，不引入 LangChain、LlamaIndex 或 LangGraph。
- 网页前端确定使用 FastAPI + Jinja2 + 少量原生 JavaScript；不引入 Node.js、React/Vue 或 Gradio。
- 配方功能确定包含直接配方、材料树、递归配方树和按 N 层展开；所有图遍历与数量计算由确定性代码完成，LLM 不参与计算。
- 多配方默认采用一个确定性主分支，其他候选折叠显示并允许用户切换；标签原料保留集合语义，不展开所有笛卡尔组合。
- 网页日志分析只处理本地玩家死亡，自动识别玩家名，支持简体中文与英文客户端日志。
- 新增独立开发者后台 `/admin`，集中展示运行状态、诊断信息、Agent/RAG/LLM 指标与 API 用量。
- 开发者后台仅允许本机访问，MVP 不实现账号登录；Docker Web 端口仅绑定 `127.0.0.1`。
- RAG 检索确定为 Qdrant 向量检索 + 标题/别名精确加权，不引入完整 BM25 索引。
- RAG 证据不足时允许 DeepSeek 使用模型常识补充，但必须单独标注“未经知识库验证”，不得与有来源事实混写。
- 每次本地玩家死亡仅调用一次 DeepSeek 生成简短建议；该路径不调用 Wiki RAG、配方工具或多轮 Agent。
- 普通交互统一显示在聊天框。死亡建议为 2–5 句短消息并自动淡出；Fabric Mod 以本地客户端聊天消息模拟另一位玩家回复。
- Fabric Mod 使用 `/pilot` 指令前缀显式触发 Wiki/配方/完整 Agent 查询；无此前缀的服务器/玩家聊天不送入 Agent。
- `/pilot` 采用自然语言 + 可选固定子命令：自然语言进入 Agent 路由，`recipe/wiki/status/help` 提供确定性快捷路径。
- 对话采用短期会话记忆：每个网页/Mod 会话保存最近约 6 轮，重启或 `/pilot clear` 后清空，不持久化完整聊天。
- DeepSeek 采用平衡用量限制：普通 Agent 最多 4 次模型往返、最终输出约 800 tokens；死亡建议单次、约 200 tokens；每日累计 200,000 tokens 后暂停自动死亡分析。
- Wiki 知识库采用显式手动更新：首次运行命令构建，平时启动只使用本地索引；`/admin` 可手动检查更新和触发增量同步。
- MVP 范围保持当前功能集，不加入库存、位置、装备或更多游戏状态能力。
- 交付优先级确定：Docker 化网页为课程必交 MVP；Fabric 26.2 Mod 为网页完成后的第二阶段加分项，不阻塞 MVP 完成。
- 自动化测试目标约 10 个高价值用例，覆盖关键算法、失败路径与安全边界；不追求大规模测试数量。
- 仓库只提交源码、构建脚本、来源/哈希元数据和小型测试夹具；不提交 Minecraft JAR、完整 Wiki 原文、模型权重、Qdrant 数据卷或完整派生配方库。
- 必须连接真实大模型，并提供模型连通性测试。
- Agent 必须接收目标、选择动作、调用至少两个工具、记录执行结果，并在失败/非法输入/超预算时停止。
- 至少一个工具须有 MCP 风格契约；课程明确不要求完整 MCP Server。
- 至少 5 个自动化测试；conda 必做，Docker 推荐；需提示词/评估样本、安全护栏、成本控制和演示证据。

## Complexity

复杂：涉及产品范围、Minecraft 版本与版本源、数据采集合法性/稳定性、RAG 质量、结构化配方模型、Agent 编排、模型供应商、交互界面、评估与课程交付。

## Candidate Capabilities

### 推荐纳入 MVP

1. **目标导向合成规划**：玩家说“我要做附魔台”，Agent 递归查询配方，生成原料总表与合成依赖步骤。这比单次配方查询更能体现 Agent 的多步工具调用。
2. **版本感知回答**：所有知识和配方绑定游戏版本/Java 或基岩版，答案明确显示适用版本，避免把不同版本资料混在一起。
3. **来源引用与置信提示**：RAG 答案附 Wiki 页面和片段；没有证据时明确说不知道，形成可演示的失败出口。
4. **死亡事件分析**：以只读方式增量监听 `latest.log`，解析新出现的玩家死亡事件，去重后触发 Agent 分析，并在网页中推送带上下文的生存/恢复建议。

### 推荐作为加分或后续能力

4. **生存进度规划**：输入目标（下界、击败末影龙、村民交易等），结合知识与配方拆解阶段性清单。
5. **物品用途与替代方案比较**：比较工具、附魔、燃料、食物、农场方案，输出有来源的权衡。
6. **故障诊断助手**：对“刷怪塔为什么不出怪”等问题先追问环境，再检索规则并给排查树。
7. **材料缺口计算**：输入已有库存，计算完成目标配方仍缺多少材料；保持纯本地、只读、低风险。

### 暂不推荐

- 当前 MVP 直接操控 Minecraft 客户端、读取存档或发送游戏指令：集成与安全成本高；先保持日志只读。
- 当前 MVP 实时识屏或完整 Mod 联动：保留接口边界，待网页版本稳定后迁移。
- 真正的 MCP Server：不是硬性要求，应优先把工具契约、调用日志和测试做扎实。

## Decision Tree / Open Questions

按依赖顺序逐项确认：

1. 产品边界：已确定当前为网页，最终演进为游戏内 Mod；网页 MVP 通过只读日志监听连接游戏状态。
2. MVP 用户旅程：知识问答、合成规划、死亡分析中哪一个作为主演示闭环？
3. 版本范围：当前收敛为 Java Edition 26.2 正式版；未来可扩展其他 26.x 稳定版。
4. Wiki 数据源与范围：已确定使用 MediaWiki API 和 Java 26.x 分类白名单；仍需落实分类清单、许可标注和更新周期。
5. 知识准确性：引用方式、无答案策略、版本冲突处理、评估指标。
6. 配方数据：来源、结构 schema、标签/替代材料、冶炼/切石等非工作台配方。
7. Agent 工具面：工具数量、契约、递归调用上限、失败语义与日志。
8. 模型路线：在线 OpenAI 兼容接口还是本地 Ollama；模型与 embedding 选择。
9. RAG 框架：轻量自建、LlamaIndex/LangChain，以及向量库选择。
10. Agent 编排：原生 tool calling 状态机还是框架编排。
11. 交互入口：Gradio、Web API + 前端，或 CLI。
12. 非功能需求：响应时间、离线能力、成本预算、数据体积与硬件限制。
13. 安全与合规：输入长度、域名白名单、日志脱敏、秘密管理、爬取节流。
14. 测试评估：至少五类自动化测试、RAG 问答集、工具选择正确率与演示样例。
15. 交付边界：conda、Docker、报告、截图/录屏及实现时间。

## Requirements (evolving)

- 单 Agent 架构，不采用多智能体。
- Wiki RAG 与配方查询是两个独立、可观测的工具能力。
- 配方工具采用 MCP 风格契约和模型 Function Calling，但不启动 MCP Server。
- 网页 MVP 只读监听用户明确选择的 `latest.log`，不修改游戏文件、不执行游戏命令。
- 检测到新的死亡事件时，Agent 主动分析死亡原因并在网页给出建议。
- 网页明确展示游戏连接状态、版本值及其来源（日志自动识别/用户手动选择）；未连接不是系统错误，也不得阻止知识与配方工具工作。
- Docker 运行日志监听时，宿主机 Minecraft 日志目录必须以只读 bind mount 显式传入；不得假设容器能直接发现宿主机进程或路径。
- 后端、Agent、知识库、配方服务和游戏事件模型不得依赖网页 UI，以便 Fabric Mod 复用。
- HTTP 用于问答、配方树、健康检查和配置查询；WebSocket 用于 Mod 连接状态、死亡事件上报及 Agent 主动建议推送。
- 网页与 Mod 应共享 API schema 和事件 schema；Mod 不存储模型密钥，不直接连接 VectorDB 或模型供应商。
- DeepSeek base URL、模型名、超时、重试和输出上限均环境变量化；不硬编码即将弃用的 `deepseek-chat` 别名。
- 向量索引元数据必须包含 embedding 模型 ID/修订版本、维度、chunker 版本和构建时间；配置变化时拒绝混用旧索引并要求重建。
- Qdrant 镜像版本必须固定，数据写入 Docker 命名卷；健康检查通过后后端才进入 ready 状态。`.venv` 开发模式也连接容器中的 Qdrant。
- 模型客户端必须支持 Tool Calls 多轮消息、Pydantic 参数二次校验、最大工具轮数和未知工具拒绝；不依赖 Beta strict mode 才能保证安全。
- Agent 状态机至少包含 `received -> deciding -> tool_running -> observing -> answered/stopped/failed`，并记录每次状态转换、工具名、参数摘要、耗时与错误类型。
- RAG 管线直接使用 MediaWiki API 客户端、切块器、sentence-transformers/等价 embedding 适配器和 Qdrant Client；每个模块提供小而明确的接口。
- RAG 为页面标题、英文名、资源 ID 和常用别名建立规范化字段；查询先识别精确实体候选，再与向量相似度按固定规则重排。
- 回答结构区分 `verified_answer`、`unverified_supplement` 与 `sources`；未经验证内容不得用于配方计算、游戏版本判定、日志事实解析或其他确定性结论。
- 若检索为空，仍明确显示“知识库未找到足够依据”，随后才可给出可关闭的模型常识补充。
- 网页以简单易用为目标，包含对话区、配方树区、游戏连接/事件状态区；Wiki 来源和 Agent 执行日志默认折叠但可展开。
- 配方工具接受目标物品、目标数量、游戏版本、最大展开深度 N 与候选配方策略；返回直接配方、树形节点、叶子材料汇总、余量和截断标记。
- 配方图构建必须处理多配方、标签材料、不可合成叶子、产出数量换算、循环检测和最大节点数；相同子图可复用但展示时不得造成无限递归。
- 默认配方排序规则必须版本化且可测试（例如普通合成优先、较少原料种类优先、资源 ID 字典序兜底）；用户选择候选配方后以选择映射重新计算全树。
- LLM 只将自然语言解析为工具参数并解释确定性结果，不生成、修改或自行计算配方数据。
- 前端通过 HTTP 提交问答与配方请求，通过原生 WebSocket 接收连接状态、死亡事件、主动建议和运行状态。
- 玩家名优先从启动/会话日志自动识别；无法可靠识别时状态功能暂停并提示原因，不把其他玩家的死亡消息误判为本地玩家。
- 死亡解析器支持 `zh_cn` 与 `en_us` 规则集，输出规范化 `DeathEvent`（玩家、原因类别、原始消息摘要、时间、置信状态）；未知消息不交给 LLM 猜测事实。
- 死亡建议请求只包含规范化死亡原因和最小必要上下文；输出设低 token 上限并要求简短、可行动、不说教。事件严格去重，不因失败自动重复调用。
- DeepSeek 不可用时前台只显示一次轻量失败状态或保持安静，详细错误进入 `/admin`；死亡路径与聊天 Agent 独立，避免影响正常游戏。
- Web 死亡建议作为聊天框临时气泡展示并自动淡出；Mod 使用客户端本地 chat HUD 注入，视觉上自然消失但可遵循游戏自身聊天历史行为。
- Mod 拦截 `/pilot ...` 后在本地调用后端，不把命令或 Agent 回复发送至多人服务器。Pilot 不监听、上传或响应普通聊天内容。
- `/pilot` 查询可输出较完整答案并调用 Wiki/配方工具，但仍设置长度上限；长配方树在聊天中给摘要，详细树通过可展开游戏界面/网页链接属于后续增强。
- 固定子命令直接调用对应后端 endpoint，不先消耗一次 LLM 路由；自然语言模式由状态机决定 `wiki_search`、`recipe_query` 或普通回答。
- 会话上下文只保存用户/助手短文本与工具结果摘要；完整 Wiki 片段、配方树、日志和死亡建议不进入长期上下文。达到轮数/token 上限时从最旧轮次裁剪。
- 用量限制按本地时区自然日累计并持久化聚合计数；达到 80% 在 `/admin` 预警。超限后无需 LLM 的 `recipe/status/help` 仍可用，计数重置操作写入审计日志。

### Developer Admin Dashboard

- **系统概览**：后端版本、启动时间、Docker/主机平台、健康/就绪状态、当前游戏版本。
- **游戏连接**：检测到的 Minecraft 进程摘要、玩家名、日志路径（脱敏）、文件 offset、最后活动时间、轮转/截断次数、WebSocket/Mod 连接状态。
- **事件流**：最近结构化游戏事件、死亡解析结果、去重状态与处理耗时；原始日志仅显示脱敏截断片段。
- **RAG 状态**：Wiki 页面/chunk 数、最后同步 revision/时间、分类覆盖、抓取失败、embedding 模型与索引 schema、Qdrant collection/point 状态。
- **配方状态**：目标游戏版本、官方清单/哈希验证状态、已提取配方/标签数量、数据构建时间与失败项。
- **LLM 与 Agent**：DeepSeek 模型、连通性、请求/错误/延迟、输入输出 tokens、估算用量、工具选择次数、平均轮数、停止原因与脱敏 trace。
- **安全/配置**：工具白名单、预算上限、超时、日志级别和配置来源；所有 secret 只显示“已配置/未配置”。
- **诊断操作**：健康检查、重新连接日志、触发增量 Wiki 同步、重建索引、导出脱敏诊断包；耗时或破坏性操作必须二次确认并记录审计事件。
- 管理端 API 校验请求来源/Host 并仅监听 loopback；不向前端返回 secret。若未来开放局域网访问，必须先增加认证与 CSRF 防护。
- 架构应将 UI、Agent 核心与游戏状态适配层分离，以便未来把网页适配层替换为 Mod 集成层。
- 不支持基岩版，也不为旧 Java 版本提供兼容承诺。
- Wiki 片段与配方数据必须保存适用版本范围；Agent 应依据日志检测或用户配置的具体 26.x 版本筛选数据。
- Wiki 采集只使用 MediaWiki API，不遍历渲染网页；保存页面 ID、revision ID、标题、来源 URL、抓取时间和版本元数据。
- Wiki 同步不属于应用启动依赖；更新任务使用 staging collection，成功验证后原子切换，失败时继续保留上一个可用索引。
- Wiki 分类范围以配置文件管理，至少覆盖方块、物品、生物、群系、机制、附魔、状态效果、结构和命令；排除基岩版专属、旧版本、开发版本及非正文命名空间。
- 配方提取器只接受官方版本清单中 release 类型的 26.x 游戏包，提取配方 JSON、物品标签和必要语言资源；校验失败或开发版本必须拒绝处理。
- 下载流程必须先获取/缓存官方版本元数据；镜像文件只有与官方 SHA-1 完全一致才可进入提取流程。校验失败的临时文件不得进入缓存或配方库。
- 下载器记录版本、实际下载源、官方期望哈希、实际哈希、重试与回退状态；日志不得记录敏感代理凭据。
- 仓库不提交 Minecraft JAR；提交提取脚本、来源元数据、官方哈希、数据 schema 与允许分发的派生结构化数据。
- 优先构成一个小而闭环、个人可完成且适合 3–5 分钟演示的产品。

## Acceptance Criteria (evolving)

- [ ] 相同问题在不同 26.x 稳定版本存在差异时，检索和配方工具返回对应版本的数据；快照内容不进入正式答案。
- [ ] 使用 `bge-small-zh-v1.5` 构建与查询中文 Wiki 向量索引，重启容器后索引仍可用；模型或切块版本不匹配时提示重建。
- [ ] 专有名词、英文名和资源 ID 查询可通过标题/别名加权稳定命中对应页面，同时保留来源 URL 与 revision ID。
- [ ] 证据不足的评估样例中，模型补充被明确分区标注且不伪造来源；确定性配方与日志事实不受模型常识覆盖。

- [ ] `/pilot` 能触发完整 Agent 工具流程，普通多人聊天不会离开客户端或进入后端。

- [ ] 连续追问可引用最近约 6 轮上下文；`/pilot clear` 与服务重启后会话清空，完整工具载荷和死亡提示不被持久化。
- [ ] 未检测到 Minecraft 运行时日志监听为关闭状态，RAG 与配方树查询仍可正常使用。
- [ ] 自动识别失败时用户可选择官方清单验证过的 Java 26.x 稳定版，并在 UI 中看到版本来源。

- [ ] Web 服务只发布到 `127.0.0.1`；从非本机来源访问管理端被拒绝，诊断操作有确认与审计记录。
- [ ] 网页 MVP 完成并通过验收后，Fabric 26.2 Mod 能复用 Agent 后端能力而不是复制知识与推理逻辑。

- [ ] 工具参数无效、资料缺失或预算超限时返回明确失败结果。
- [ ] 至少 5 个自动化测试通过，并有真实模型连通性证据。

- 仓库当前无 `.trellis` 目录，因此用 `tasks/07-04-minecraft-pilot-agent/` 作为轻量任务目录。

- 版本策略决策（最新）：本次固定支持 Java 26.2 正式版；保留版本化 schema 作为未来扩展点，前述“全部 26.x”不再是本次验收承诺。
- Wiki 数据源决策：`https://zh.minecraft.wiki/api.php`；采集器应支持分页、限速、重试、断点续传、响应缓存和基于 revision ID 的增量更新。
- Wiki 范围决策：使用 Java 26.x 相关分类白名单；产品承诺表述为“覆盖 Java 26.x 核心游戏知识”，不承诺任意 Minecraft 话题全覆盖。
- 配方来源决策：以官方发布的各 Java 26.x 稳定版原版数据包为唯一事实源；需记录版本 ID、下载/本地来源、官方哈希和提取时间。
- 游戏包下载决策：官方 CDN 优先，国内镜像回退；镜像只提供字节传输，官方清单与哈希仍是信任根。镜像 URL 应配置化，默认白名单且支持禁用。
- 运行版本决策：日志自动识别并允许手动覆盖；无游戏进程时关闭监听、保留 RAG 与配方能力，形成明确的降级运行模式。
- 平台/交付决策：Docker 化网页与 VectorDB，优先验证 macOS；跨平台仍需分别处理日志目录挂载与权限，Docker 不等同于自动跨平台日志发现。
- 产品演进决策：Phase 1 网页 MVP；Phase 2 Fabric API 26.2 Mod。官方 Minecraft 26.2 发布于 2026-06-16，Fabric 官方站点已列出 26.2 支持。
- 交付决策：Phase 1 网页是课程必交；Phase 2 Mod 是加分项，失败或延期不影响网页 MVP 验收。
- 测试决策：约 10 个关键自动化测试 + 独立真实模型 smoke test；优先关键链路质量而非数量。
- 数据分发决策：source/build-scripts/metadata/fixtures only；完整知识、配方和向量数据由用户在本地显式构建，并加入 `.gitignore`。

## Technical Approach

- **Runtime**：Python 3.12、FastAPI、Pydantic、Jinja2、原生 JavaScript；pip + `.venv`，Docker Compose 运行 FastAPI 与 Qdrant。
- **LLM**：DeepSeek OpenAI 兼容 API，模型名环境变量化，轻量自建 Agent 状态机与 Function Calling。
- **RAG**：中文 Minecraft Wiki MediaWiki API 分类白名单 → 清洗/切块 → `bge-small-zh-v1.5` → Qdrant；dense 检索加标题/别名 boost。
- **Recipes**：Mojang 官方 26.2 release 清单与 SHA-1 为信任根 → 下载/镜像回退 → 原版数据提取 → SQLite → 确定性 N 层配方树。
- **Game state**：网页阶段只读监听 `latest.log`，自动识别 26.2 与本地玩家，解析 `zh_cn`/`en_us` 死亡消息；未检测到游戏时关闭监听。
- **Clients**：Phase 1 网页聊天与 `/admin`；Phase 2 Fabric 26.2 Mod 通过 localhost HTTP/WebSocket 复用后端，所有消息仅本地显示。
- **Observability**：结构化 trace、tokens/预算、RAG/配方构建状态、游戏进程/日志 offset、WebSocket 连接与脱敏诊断。

## Decision (ADR-lite)

**Context**：课程要求真实模型、单 Agent、至少两个工具、RAG、MCP 风格契约、失败出口、测试、成本与本地交付；产品长期希望进入游戏内。

**Decision**：先交付 Docker 化网页 MVP，以 Wiki RAG、确定性配方树、死亡日志建议为三大能力；后端采用稳定 API/事件契约，第二阶段添加轻量 Fabric Mod。

**Consequences**：网页可先独立验收，Mod 不复制 AI 逻辑；Docker 仍需显式挂载宿主机日志。明确拒绝 conda 会与课程硬性环境条款冲突，可能产生扣分，报告必须如实说明。

8. **M8 实验报告**：严格按照 `docs/实验报告提交及评分标准.md` 撰写完整项目报告，覆盖 13 个规定章节、仓库/提交历史、架构、真实模型、Agent/工具、课程知识点、Docker、测试证据、失败修正、安全成本、AI 编程过程与改进方向。
9. **M9 加分阶段**：Fabric 26.2 Mod，客户端 `/pilot`、本地 chat HUD、结构化游戏事件与后端断连降级。
- Mod 集成决策：轻量 Fabric 客户端 + 本地 HTTP/WebSocket 后端。网页与 Mod 复用服务契约，差异仅在输入事件源和展示适配层。
- 后端决策：Python 3.12、FastAPI、Pydantic；HTTP + WebSocket；OpenAPI 作为网页与 Fabric Java DTO 的契约来源。
- 环境决策：仅使用 pip + `.venv` 与 Docker，永不使用 conda。课程明确要求 conda 路径，因此报告必须如实说明该偏差，不能声称满足 conda 验收项。
- LLM 决策：DeepSeek 在线 OpenAI 兼容 API，密钥来自 `.env`；模型名配置化。官方文档确认支持 Tool Calls，且旧 `deepseek-chat` 别名计划于 2026-07-24 弃用。
- Embedding 决策：本地 `BAAI/bge-small-zh-v1.5`，容器内推理，接口可替换；MVP 不引入额外 embedding API。
- VectorDB 决策：Qdrant 仅通过 Docker Compose 运行，固定镜像版本与持久化卷；不使用 Homebrew 安装 Qdrant。
- Agent/RAG 框架决策：轻量自建状态机与检索管线，直接调用底层 SDK；不采用 LangChain、LlamaIndex 或 LangGraph。
- Web UI 决策：FastAPI + Jinja2 + 原生 JavaScript，简单易用、无 Node 构建链；对话/配方/状态三块布局。
- 配方决策：直接配方、材料树、完整/限深配方树均由代码计算；支持 N 层复杂度，LLM 仅负责 Function Calling 与自然语言说明。
- 多配方决策：确定性默认主分支 + 折叠候选 + 用户可切换；标签原料作为集合节点处理。
- 日志范围决策：仅本地玩家；自动识别玩家名；支持 `zh_cn`/`en_us`；未知死亡不猜测。
- 死亡建议决策：每次死亡单次 DeepSeek 调用，独立于 RAG/配方/多轮 Agent；简短、低打扰、失败不刷屏。
- 交互决策：所有用户可见输出进入聊天框；死亡建议短且淡出；Mod 通过本地 `/pilot` 显式查询，Pilot 不读取普通聊天。
- 命令决策：自然语言 + 可选子命令；固定子命令绕过 LLM 路由，自然语言保留 Agent 体验。
- 记忆决策：短期、会话级、约 6 轮、仅保留工具摘要；不持久化聊天，死亡事件不加入对话记忆。
- 成本决策：平衡限制（Agent 4 往返/约 800 输出 tokens，死亡单次/约 200 tokens，每日 200k tokens）；80% 预警，阈值环境变量化。
- Wiki 更新决策：显式首次构建 + `/admin` 手动检查/增量同步；不在启动时或后台定时自动访问 Wiki，更新失败保留旧索引。
- 运维界面决策：新增 `/admin` 开发者后台，兼顾实时诊断、课程演示证据、用量统计与安全审计。
- 管理端访问决策：loopback-only、无登录、最小复杂度；secret 永不回传，诊断变更需确认。局域网访问不在 MVP 范围。
- RAG 检索决策：dense vector + title/alias exact boost；MVP 不实现完整 BM25 混合检索。
- 证据策略决策：允许模型常识补充，但必须标注“未经知识库验证”并与有引用答案结构化分离。
